# Fork Overlay Workflow

Схема работы с форком апстрим-проекта, при которой `main` остаётся **тонким зеркалом апстрима**, а все личные доработки живут на отдельных ветках. Цель — быстро и безболезненно тянуть обновления из апстрима, не разгребая конфликты с накопленными личными коммитами.

Документ самодостаточный: его можно положить в любой форк и применить.

---

## Принцип

- `main` = строгое зеркало `upstream/main`. Никаких личных коммитов. Только fast-forward от апстрима.
- Каждая личная доработка живёт в **отдельной ветке** от `main`. Одна тема — одна ветка.
- Ветка `deploy` (можно назвать `prod`/`local`/как угодно) — рабочая сборка, с которой запускается софт. Это `main` + все личные ветки, наложенные сверху merge или rebase.

```
upstream/main ──●─────────●─────────●──── (апстрим движется)
                │         │         │
fork/main     ──●─────────●─────────●──── (ровно догоняет апстрим, ff-only)
                │         │         │
feat/A        ──○         ○         ○──── (rebase на новый main при апдейте)
fix/B         ──○         ○         ○────
local/docker  ──○         ○         ○────
                │         │         │
deploy        ──●─────────●─────────●──── (main + все ветки, force-push разрешён)
```

---

## Типы веток

| Префикс | Назначение | Идёт ли в апстрим |
|---|---|---|
| `feat/*`, `fix/*`, `chore/*` | Изменения, потенциально интересные апстриму. Маленькие, по одной теме, с чистой историей. | Да — открываем PR. |
| `local/*` | Личное: Dockerfile, justfile, CI, приватные доки, бизнес-специфичные конфиги. | Нет. |
| `local/meta` | Спецветка под мета-файлы форка: `BRANCHES.md` и любые внутренние инструкции. Содержит **только** доки, никаких правок кода. | Нет. |

Главное правило именования: **одна ветка = одно логическое изменение**. Если сложно объяснить ветку одной фразой — её надо разделить.

### Где живёт реестр веток

`BRANCHES.md` (и любые другие мета-файлы про сам форк) **нельзя класть в `main`** — он затрётся при первом же `merge --ff-only upstream/main`. Поэтому реестр живёт на отдельной ветке `local/meta`, ответвлённой от `main` и содержащей только мета-файлы. `local/meta` мержится в `deploy` наравне с остальными `local/*`, поэтому в собранной копии файл доступен. При обновлении с апстрима ветка просто ребейзится на новый `main` (конфликтов почти не бывает — в апстриме этих файлов нет).

---

## Жёсткие правила

1. **Никогда** не коммить в `main`. `main` обновляется только через `git merge --ff-only upstream/main`.
2. Каждая личная ветка ответвлена от **актуального `main`** (= `upstream/main`).
3. Каждая ветка автономна: не зависит от другой личной ветки. Если зависит — это знак, что они должны быть одной веткой.
4. `deploy` — **технологическая** ветка, её можно force-push'ить. На `deploy` никто не пишет вручную, она пересобирается из `main` + ветки.
5. Включить `git config rerere.enabled true` — один и тот же конфликт при ребейзе не придётся решать дважды.

## CI в форке

Апстрим обычно тащит свой CI/CD (`.github/workflows/*`, GitLab CI и т.п.). В форке он чаще всего **бесполезен или вреден**: гоняет чужие пайплайны без секретов, шумит уведомлениями, расходует лимиты Actions.

Правило:

- **На `main` CI отключён.** В форке `main` = зеркало апстрима, но запускать в нём пайплайны не нужно. Самый простой способ — **GitHub → Settings → Actions → Disable actions** для всего репо. Это глобально, не требует правок файлов и не ломает зеркальность.
- **Свой CI добавляем только в `deploy`** — и только по запросу, под конкретную задачу (билд образа, прогон тестов на собранной сборке, авто-деплой). Workflow-файлы коммитим в отдельную ветку `local/ci-deploy` (или похожую) и мержим в `deploy`. Тогда CI бежит только на ветке `deploy` и не цепляет ни `main`, ни личные feature-ветки.
- Если для какого-то workflow нужно ограничение по веткам — прописать его в самом yml через `on.push.branches: [deploy]`.

Так апстрим-CI не будет неожиданно срабатывать после `merge --ff-only`, а свой пайплайн появится строго там, где он нужен.

## Цикл обновления с апстрима

```bash
# 1. Подтягиваем апстрим
git fetch upstream

# 2. main fast-forward
git checkout main
git merge --ff-only upstream/main
git push fork main           # ff, force-push не нужен

# 3. Ребейзим каждую личную ветку на свежий main
for branch in feat/A fix/B local/docker local/config; do
    git checkout "$branch"
    git rebase main
    # если конфликт — резолвим, rerere запомнит
    git push -f fork "$branch"
done

# 4. Пересобираем deploy
git checkout deploy
git reset --hard main
for branch in feat/A fix/B local/docker local/config; do
    git merge --no-ff "$branch" -m "deploy: include $branch"
done
git push -f fork deploy
```

Если апстрим вмержил твою ветку — удаляешь её, на следующем `merge --ff-only` изменения приедут в `main` сами.

## Как ввести схему в уже разошедшийся форк

1. Найти коммиты, ушедшие вперёд `upstream/main`:
   ```bash
   git fetch upstream
   git log upstream/main..main --oneline
   ```

2. Прогнать patch-id скан — какие из них уже приняты апстримом и могут быть дропнуты:
   ```bash
   declare -A want
   for sha in $(git log upstream/main..main --pretty=%H); do
     pid=$(git show "$sha" | git patch-id | awk '{print $1}')
     want["$pid"]="$sha"
   done
   while read -r usha; do
     pid=$(git show "$usha" 2>/dev/null | git patch-id 2>/dev/null | awk '{print $1}')
     [ -n "$pid" ] && [ -n "${want[$pid]:-}" ] && \
       echo "MATCH local=${want[$pid]} upstream=$usha"
   done < <(git log upstream/main --pretty=%H)
   ```

3. Сгруппировать оставшиеся коммиты по темам. Для каждой группы — отдельная ветка от `upstream/main`:
   ```bash
   git checkout -b feat/X upstream/main
   git cherry-pick <sha1> <sha2>
   ```
   Для крупных «кухонных моек» (один коммит, много файлов разных тем) — cherry-pick во временную ветку, затем разнести файлы по тематическим веткам через `git checkout <tmp> -- <files>`.

4. Сделать тег-страховку:
   ```bash
   git tag main-backup-$(date +%Y-%m-%d) main
   ```

5. Сбросить `main` к `upstream/main` и собрать `deploy`:
   ```bash
   git checkout main
   git reset --hard upstream/main

   git checkout -b deploy main
   for b in <свои ветки>; do git merge --no-ff "$b"; done
   ```

6. Force-push только `main` (один раз, чтобы переписать историю форка) и `deploy`. Остальные ветки — обычным push.

7. Создать ветку `local/meta` от свежего `main` и положить туда `BRANCHES.md` (и эту инструкцию, если хочешь, чтобы она жила в репе). Включить `local/meta` в сборку `deploy`.

## BRANCHES.md — шаблон реестра

```markdown
# Active personal branches

| Branch | Type | Purpose | Upstream PR |
|---|---|---|---|
| feat/X | feat | one-line описание | #123 (open) |
| fix/Y | fix | one-line описание | merged → можно удалять |
| local/docker | local | Dockerfile, compose, justfile | n/a |
| local/config | local | локальные дефолты конфига | n/a |

## Recipe
Запуск/деплой собирается из ветки `deploy` = `main` + все ветки выше через `git merge --no-ff`.
```

## Когда удалять ветку

- Апстрим вмержил твой PR → ветка больше не нужна. Удалить локально и из `deploy`.
- Изменение перестало быть актуальным → удалить ветку и убрать её из сборки `deploy`.
- Ветка превратилась в постоянный конфликт с апстримом → пересмотреть: либо переписать с нуля от свежего `main`, либо отказаться.

## Антипаттерны

- ❌ Коммитить что-либо в `main`.
- ❌ Делать ветку, зависящую от другой личной ветки.
- ❌ «Мегакоммиты» (25 файлов, +2000 строк) — нельзя ревьюить, нельзя PR-ить, нельзя выборочно откатить.
- ❌ Разрабатывать прямо в `deploy`. `deploy` всегда производна от `main` + ветки.
- ❌ Force-push в `main`. Если очень надо — только в момент первичной перестройки схемы (см. «Как ввести схему»), и с предварительным тегом-страховкой.

---

## Минимальный чек-лист «работает ли схема у меня»

- [ ] `git log upstream/main..main` — пусто.
- [ ] Каждая личная ветка ответвлена от `main` и описана в `BRANCHES.md` (на ветке `local/meta`).
- [ ] `deploy` собирается одним скриптом из `main` + ветки.
- [ ] `git config --get rerere.enabled` → `true`.
- [ ] Обновление с апстрима занимает один цикл `fetch → ff main → rebase ветки → пересобрать deploy`.
